# 编码准则

与 agent-core 中 Simplicity First、Surgical Changes、不兜底互补；
审查时由 code-review skill 对照。

## 状态与封装

尽可能不增加持久的类/实例状态或模块级可变状态：
- 优先通过参数传递数据
- 考虑用闭包封装局部状态
- 降低对象或模块的状态复杂度

只使用一次的逻辑倾向于不单独抽取，或封装为闭包：
- 避免过度抽象
- 一次性逻辑考虑内联
- 需抽取时优先局部函数或闭包

## API 精简

最小化对外公开接口表面积：
- 只暴露必要的 public / pub / export 成员
- 移除仅内部使用的对外成员
- 合理默认值减少配置项
- 保持 API 一致、简洁

## 复杂情况

难以组织的复杂逻辑可暂跳过、后续再处理；优先保证整体结构清晰。

## 运行时错误处理

- 非预期情况优先打日志，**不**为兼容加兜底
- 日志附带上下文与关键参数
- 仅在开发期调试或数据无法继续时考虑断言/抛异常等强制中断

语言具体风格见项目 `AGENTS.md`、各模块 README。新增 feature 的测试要求见项目测试规范。

## 适用时机

编辑项目源码、code 阶段产出、code-review 审查时对照本 skill。纯文档讨论或只读分析不适用。

## 反合理化

| 借口 | 反驳 | 级别 |
|------|------|------|
| "先能跑再加封装" | API 精简与状态封装是设计的一部分，不是事后重构项。 | 重要 |
| "加个兜底更稳" | 不兜底：非预期打日志暴露，不为兼容吞异常。 | 重要 |
| "抽象一下以后复用" | Simplicity First：只用一次的逻辑不单独抽取。 | 重要 |
| "这规则太细，这次例外" | 例外不经审查就是技术债。由 code-review 判定是否放宽。 | 重要 |
